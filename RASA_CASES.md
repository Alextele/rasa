# Rasa Bot — Кейс

---

## Кейс — Rule-Based Dialogue Agent с пользовательским управлением обучением

**Что это:** альтернативный подход к построению чат-бота без языковой модели (LLM). Вместо генерации текста — **классификация намерений + детерминированное управление диалогом + шаблонные ответы**. Всё содержимое бота (сценарии, вопросы, ответы, сущности) управляется пользователем через интерфейс и хранится в базе данных — модель пересобирается автоматически при каждом изменении.

---

### Чем отличается от RAG

| Аспект | RAG / LLM | Rasa |
|---|---|---|
| Понимание запроса | LLM интерпретирует свободный текст | Классификатор определяет **intent** |
| Ответ | Модель генерирует текст | Заранее написанный шаблон или action |
| Управление диалогом | История сообщений в промпте | **Dialogue state machine**: слоты, политики, правила |
| Предсказуемость | Низкая — генерация непредсказуема | Высокая — детерминированные правила |
| Обновление знаний | Обновить контекст / векторную БД | Добавить примеры → пересобрать модель |
| Применимость | Открытые знания, FAQ, консультации | Структурированные сценарии, формы, бизнес-процессы |

---

### Как работает стек

#### 1. NLU-пайплайн: понимание реплики

Каждое сообщение пользователя проходит через **NLU (Natural Language Understanding) pipeline**:

- **Токенизация** (`WhitespaceTokenizer`) — разбивка текста на токены
- **Векторизация** (`CountVectorsFeaturizer`) — преобразование токенов в числовые признаки
- **DIETClassifier** (Dual Intent and Entity Transformer) — многозадачная модель, которая одновременно:
  - классифицирует **intent** (намерение): к какому классу относится фраза — `greet`, `ask_weather` и т.д.
  - выполняет **NER** (Named Entity Recognition): извлекает именованные сущности — например, `city = "Москва"` из фразы «погода в Москве»
- **EntitySynonymMapper** — нормализует синонимы сущностей: «Питер», «СПб», «Санкт-Петербург» → одно значение

Модель обучается исключительно на примерах из `data/nlu.yml`. Она не «понимает» язык как LLM — она обобщает паттерны из размеченных фраз. Сущности размечаются через BIO-теги: `[Москве](city)`.

#### 2. Slots: память диалога

После NLU-пайплайна извлечённые сущности записываются в **слоты** — переменные состояния диалога. Слот `city: type: text` с маппингом `from_entity` автоматически принимает значение из сущности `city`. Пока слот не заполнен, бот не располагает нужными данными для ответа — это основа **slot filling** паттерна: бот уточняет недостающее, пока не накопит всё необходимое.

#### 3. Dialogue Management: управление сценарием

Rasa имеет три уровня управления диалогом с разным приоритетом:

- **RulePolicy** (наивысший приоритет) — детерминированные правила из `data/rules.yml`. Если условие по слотам выполнено → действие однозначно. Не обучается, не может «забыть». Используется для критических сценариев — всегда срабатывает предсказуемо.
- **MemoizationPolicy** — запоминает точные последовательности из stories и воспроизводит их дословно при совпадении.
- **TEDPolicy** (Transformer Embedding Dialogue) — ML-политика, обученная на stories. Предсказывает следующее действие на основе истории диалога (`max_history: 5`). Покрывает нелинейные сценарии, которые не описаны правилами явно.

В текущем проекте реализован **conditional branching** через rules: если `city = null` → спросить город, если `city` заполнен → вернуть погоду.

#### 4. Custom Action Server: бизнес-логика

Когда политика решает выполнить нестандартное действие (`action_get_weather`), Rasa отправляет HTTP-запрос на отдельный **action server** (`rasa/rasa-sdk`, порт 5055). Action server выполняет Python-код:

- читает слоты через `tracker.get_slot("city")`
- обращается к внешнему API, базе данных — любая бизнес-логика
- формирует ответ через `dispatcher.utter_message()`

Action server — независимый Docker-контейнер, масштабируется отдельно от NLU. `Tracker` предоставляет полный доступ к состоянию диалога: слоты, историю событий, последний intent.

### Архитектура с пользовательским управлением

Ключевая идея продукта: **пользователь не редактирует YAML-файлы вручную**. Он управляет ботом через интерфейс — добавляет интенты, примеры фраз, сущности, правила, шаблоны ответов. Всё хранится в базе данных. Модель пересобирается автоматически.

```
┌─────────────────────────────────────────────────────────────────┐
│                        Admin Interface                          │
│   Интенты / Примеры фраз / Сущности / Правила / Ответы          │
└──────────────────────────┬──────────────────────────────────────┘
                           │ CRUD
                           ▼
                    ┌─────────────┐
                    │  Database   │  (PostgreSQL / MongoDB)
                    └──────┬──────┘
                           │ Событие изменения
                           ▼
               ┌───────────────────────┐
               │   Training Pipeline   │
               │  DB → YAML-генерация  │  nlu.yml / rules.yml /
               │  → rasa train         │  domain.yml / stories.yml
               └──────────┬────────────┘
                           │ новая модель (*.tar.gz)
                           ▼
               ┌───────────────────────┐
               │     Rasa Server       │  PUT /model  (hot swap —
               │   порт 5005           │  без перезапуска контейнера)
               └───────────────────────┘
```

**Hot model swap** — Rasa поддерживает замену активной модели через REST API (`PUT /model`) без остановки сервера. Пользователь публикует изменение → pipeline генерирует YAML → обучает модель → загружает её в работающий инстанс.

### Что реализовано в проекте

| Компонент | Статус |
|---|---|
| NLU pipeline (DIETClassifier + NER) | есть |
| Slot `city` + маппинг `from_entity` | есть |
| Rule-based dialogue (RulePolicy) | есть |
| Custom action server | есть (заглушка — реальный API не подключён) |
| TEDPolicy + MemoizationPolicy | подключены, stories не написаны |
| Admin interface | **нет** |
| БД как источник обучающих данных | **нет** |
| Автогенерация YAML из БД | **нет** |
| Автоматический retrain + hot swap | **нет** |
| NLU Fallback (низкий confidence → отказ) | **нет** |
